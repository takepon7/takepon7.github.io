# PLAN.md — `gitai`（擬態）/ 鑑定士

> 作業コードネーム `gitai`。AIを騙す非同期ステルス・ドローイングゲーム。
> マスコット兼ブランド装置＝自信過剰な美術鑑定士「鑑定士」。
> ※名称はプレースホルダ。後でリネーム可。

このドキュメントは「何を作るか」ではなく**「なぜこの設計なのか」**を記録する。
細部の実装は `KICKOFF.md` の各フェーズプロンプトに委譲する。
ここの設計判断ログ(§1)が、目標①（設計判断で評価される）のポートフォリオ本体である。

## 0. コア・テーゼ

### 何を作るのか
お題 `(X→Y)` が出る。プレイヤーは**明確な正体Xを持つ素体**（例：リンゴ）に描画ツールで絵を描き、**ターゲットY**（例：野球ボール）に擬態させる。視覚モデル（CLIP/SigLIP）が「YとXのどちらに見えるか」を判定し、その判定でスコアが決まる。**リアルタイム対戦なし・AI判定・完全非同期。**

### なぜこの賭けなのか（めっちゃカメレオンからの抽出）
めっちゃカメレオン（2人・約2か月・16日で1000万本・サーバー費ゼロ）の再現可能な本質は、**ジャンル（対戦かくれんぼ）ではなくエンジン**である：

> **［誰もが知るルール（学習コストゼロ）］ × ［創造をプレイヤーに委譲する1機能］ + ［判定/暴露の瞬間］ → 観戦＝参加が一致**

ジャンルのクローンは我々のスタック（Phaser+Capacitor / ソロ）にとって自殺行為：リアルタイムnetcodeはソロ最難所、バズったらサーバー費が破産させる、本家＋AIクローン勢とのレッドオーシャン。

そこで**「判定の瞬間」を担う人間の鬼をAIに置換**する。これで netcode とサーバー費が消え、しかも「AIを騙す」題材自体が2026年の文脈に乗る。AnthropicのAPIをアプリに組み込んだ実績（biz-english-master 等）が、本家にも他インディーにも真似しづらい**非対称な武器**になる。

### 期待値の前提（忖度なし）
めっちゃカメレオンは**バイラルの宝くじの当選結果**であり、再現確率は低い。「2人で数十億」を期待値の基準にしない。ただし本家の教訓——(1)創造の委譲 (2)マーケのコアループ内蔵 (3)跳ねても死なないコスト設計 (4)自分の資産流用で速く出す (5)低価格/低摩擦——は当選確率は上げないが**下限（フロア）を確実に上げる**。この設計はその5つを全て構造に埋め込む。

## 1. 設計判断ログ（load-bearing decisions）

各判断は「なぜ」と「これを外すと何が死ぬか」をセットで記録する。Claude Code はこれらを**不変条件として扱い、勝手に緩めてはならない**。

### 決定A：勝利条件は「消す」ではなく「化けさせる（陽性）」
「AIに見つからないように消す」を勝利条件にすると、**最適解が白紙/ノイズになって即死**する（無技術・無笑い）。素体Xから始めることで、何もしなければAIは「X」と答える＝負け。プレイヤーは能動的に知覚を上書きせざるを得ない。**白紙チートを構造的に除去**し、「擬態＝正体を隠す」という本家の魂は保つ。

### 決定B：判定オラクルを2つに物理分離する
- **スコア用＝自前のSigLIP/CLIP（決定的・無料・校正済み確信度）。** チャット系VisionのVisionは非決定的でスコアが揺れる＝ランキングに毒。譲れない。
- **笑い用＝LLM Vision（鑑定士の珍コメント）。キャッシュ＋ヒーロー限定。**

DDD的に言うと：**演出（コメント）層はドメインではない。** 判定の権威を一切持たない化粧であり、Scoring コンテキストの**外**に置く。これは「LLMに判定させてはいけない」を設計言語で表現したもの。

### 決定C：LLMは判定せず「演じる」
現代のVision LLMはCLIPより賢いので、CLIPを騙せた絵を見せると**正しく見抜いてしまい**、「AIが自信満々に騙される」という中核の笑いが死ぬ。よってLLMには画像判定をさせず、**CLIPの判定を「真実」として渡し、それを信じきった鑑定士として演じさせる**。判定と笑いが完全一致し、LLM暴走事故も消える。

### スコアリングのキーストーン
- **問題0：CLIP cosineは確率ではない。** 0.15〜0.35の狭帯に固まる。温度τ付きsoftmaxが必須（CLIP本来値は約100だが、**τはゲームフィールの最重要ノブ**——高いと判定が二値化して上達フィードバックが消える。シードでスコア分布を見ながら20〜50程度に調整する**チューニング対象**）。
- **致命傷1：タイポグラフィック攻撃。** 画像に「BASEBALL」と書くとCLIPは読んで分類する。OCR前段ガードで、ラベル語との一致を検出したら**ハード0点**（減点ではなく殺す）。99%の人がこの穴を見ていない。
- **キーストーン：ペアごとパーセンタイル正規化。** ペアによって到達可能な最大スコアが違う（リンゴ→ボールは高く出る、猫→哲学書は低い）ので、生スコアでのグローバルランキングは不公平。生スコアを**そのペアのシード分布の中での百分位**に変換する。パーセンタイル出力は定義上 [0,1] に一様分布するので、**ペア間不公平（致命傷2）とスコア分布崩壊（致命傷3）を1手で同時に殺す**。これがシステムの背骨。
- **効率は核スコアから外す。** ストローク数/時間を核に混ぜると「誰が一番AIを騙したか」の純粋信号が濁る。効率は独立ラダーに隔離。
- **ハードネガティブはペア定義の一部として凍結。** Xにも Yにも近い「もっともらしい誤認候補」をK個。提出ごとに変えると分母が変わりスコアが比較不能になる。
- **描画ドメインのプロンプト整形。** `"a photo of {X}"` ではなく `"a crude drawing of {X}"` 等でアンサンブル。外すと素人画の判定精度が体感で大きく落ちる。

### お題プールのキーストーン
- **「良いお題」＝シード分布の幅（spread）が広いお題。** 平均が高い/低いではなく**分散**。幅が広い＝下手と上手で割れる＝技術介入が効く＝バカ↔天才のスペクトラム。本家の「画力で千差万別」をデータで担保する唯一の指標。
- **a-priori難易度（タグ距離）は候補絞り込み専用。** プレイヤーに見せる/ランキングに使う本物の難易度は**シード分布の測定値**（p50・spread）。priorを見せるとタグ誤差がUXの嘘になる。
- **生成（高コスト）と採点（安価）を2層分離保存。** SeedAsset（画像・永久保存・二度と作らない）と SeedScores（モデル毎・安価に再計算可能）。シーズンでモデルをローテーションするとき、保存済み画像を新モデルで**再エンコードするだけ＝有料画像生成API呼び出しゼロ**。これが「跳ねても破産しない」をコンテンツ更新でも維持する鍵。
- **色は難易度に入れない。** プレイヤーは全部塗り直すので元の色は難易度をほぼ規定しない。色はテーマ分類（「食べ物週間」）と見栄えのタグに格下げ。

### 演出層のキーストーン
- **3層構成：Layer0 CLIP判定（事実）/ Layer1 テンプレバンク（大多数・LLM呼び出しゼロ）/ Layer2 LLM Vision（ヒーロー/オンデマンドのみ）。**
- **笑いの大半はLLMを呼ばない。** 中核の笑いは「自信満々の判定 × 実際の絵」の**カード上の並置**から生まれる。猫の絵の隣に「野球ボール・95%」が出れば、コメントが画像を理解せずとも視覚的にオチる。LLMが足すのは画像特化リフという**より面白いデルタ**だけで、ヒーロー時のみ正当化。
- **コメントは高温度OK（スコアの決定性絶対とは対比）。** 同じ絵が別コメントでもスコアに影響しないので、毎回違う方が切り抜きの鮮度が出る。ただし**ヒーロー投稿のコメントは鋳造後に凍結**（その切り抜きのアイデンティティ＝正典）。
- **茶化す対象は「絵」のみ、「人」は絶対に対象にしない。** 行儀ではなく安全：本人罵倒ゲームは自己批判を強化して人を傷つけうる。設計で線を引く。

## 2. アーキテクチャ

### 2.1 DDDをどこに効かせ、どこに効かせないか（重要な設計判断）
あなたの sengoku-s の流儀「ECSは戦闘、DDDはメタ進行だけ」と同型の判断を適用する。

- **DDDを効かせる：** バックエンドのドメイン3コンテキスト（採点・お題・競争）。ここには真の不変条件がある。
- **DDDを効かせない（意図的に）：**
  - **推論モデルの実行**は副作用を持つ外部能力。Domain層に置けば、あなたのスキルが警告する「DDD風の手続き型コード」になる（「画像をCLIPに通す」に守るべきドメイン不変条件はない）。→ **Infrastructure のアダプタ**として、Domainの`JudgeModel`ポートの裏に隠す。
  - **クライアント（TS/Phaser/Capacitor）**はゲーム客体。UI/描画であってDDDではない。

この線引き自体が目標①のポートフォリオ価値：**「採点ポリシーはドメイン、推論はポートの裏、演出は権威ゼロでドメイン外」**と一文で説明できること。

### 2.2 境界づけられたコンテキスト

**Scoring（採点）** — 心臓部
- 不変条件：`同一(画像, pair_id, ref_version) → 同一score`（再採点監査・チート検証の前提）／OCRチート→ハード0／効率は核スコアに非干渉。
- Aggregate：`Submission`（プレイヤーの一投：画像＋ストロークログ＋導出スコア）。小さく保つ。
- Domain Service：`ScoringPolicy`（raw→percentile→score。`JudgeModel`ポートと当該ペアのシード分布に依存）。
- VO：`Score` `Percentile` `Confidence(Cy,Cx)` `RefVersion(model_version+template_set_id+tau+seed_snapshot+freeze_date)`。プリミティブ・オブセッション回避。
- ポート（Domain I/F、実装はInfra）：`JudgeModel` `OcrScanner`。

**Puzzle（お題）**
- 不変条件：採用ペアは `spread >= floor`／`p90 >= impossible_floor`（不可能でない）／`p10 <= trivial_ceiling`（自明でない）／ペア定義 `(X,Y,hard_negatives[])` は凍結。
- Aggregate：`Pair`（root。素体X/ターゲットY/ハードネガティブ/シード分布参照を保持）、`DailyPuzzle`（root。Pair＋凍結された`ref_version`＋その日の分布スナップショット境界）。
- Domain Service：`CurationPolicy`（spreadゲート判定）。
- ポート：`ImageGenerator`（GPTシード生成）`SeedImageStore`（画像永久保存）`Tagger`（LLMタグ付け）。

**Competition（競争）**
- 不変条件：シーズン内でジャッジモデル不変／報酬はコスメのみ（リアルマネー取引なし）／ランクスコアはサーバー権威。
- Aggregate：`Leaderboard`（デイリー/フレンド/効率/面白い、の各ラダー）、`Season`（モデルピン留め＋リセット境界。巨大化させずポリシー寄りに保つ）。
- ポート：`LeaderboardRepository` `FriendGraph` `PushNotifier`。

**Presentation/演出（コンテキストではない）**
- 鑑定士コメント層。**ドメインの権威ゼロ**。`CommentActor`（LLM）＋`Moderation`＋テンプレバンク。Scoringの外。

### 2.3 技術スタック（資産流用マップ）
- **クライアント：** TypeScript + Vite + Capacitor（sengoku-s と同一→流用）。描画は**素のHTML5 Canvas 2D**（ペイントツールの正しいプリミティブ。Phaserはv1では過剰、必要になってから）。
- **採点サービス：** Python + FastAPI + ONNX Runtime + open_clip/SigLIP + OCR（Tesseract等）。ステートレス推論。MLエコシステム都合でPythonは不可避。
- **データ：** SQLite を runtime backbone に（あなたの autonomous pipeline の流儀と一致）→ スケールしたらPostgres。pairs / seed_scores / daily_puzzles / submissions / leaderboards。
- **シード生成＆タグ付け：** Python スクリプト（GPT画像生成パイプライン＝sengoku-s 流用）。
- **シェアカード：** Python + Pillow + Noto Sans CJK JP（既存のインフォグラフィックカード生成を流用）。
- **鑑定士コメント：** Anthropic API（Claude）。既存の組み込み知見を流用。

netcodeゼロ・3Dゼロ。バックエンドは薄い採点サービス＋永続層。**ソロで素直に作れる構成。**

正直なトレードオフ：**クライアントTS + バックエンドPythonの2言語**になる。これは避けられない（ML はPython一択）。コンテキスト切替コストは受け入れる。

## 3. データモデル（要点スキーマ）

詳細な型定義は KICKOFF 各フェーズに置く。ここは全体像。

```
ObjectCatalog                      # 物体カタログ
  object_id
  # (A) ジャッジ向け
  canonical_label, aliases[]       # aliasesはOCRガードにも使う
  # (B) 難易度・キュレーション向け
  shape_vec[5]                     # [丸み,細長さ,平面性,手足性,細部密度] 各0..1
  malleability                     # 素体としての化けやすさ 0..1（X側ドライバ）
  evocability                      # 描線で想起させやすさ 0..1（Y側ドライバ）
  category                         # food/tool/animal/... 週替わりテーマ用
  dominant_colors[]                # テーマ/見栄えのみ。難易度に使わない
  # (C) 運用
  status(active|retired), source(llm_proposed|human_curated)

TextEmbeddingCache                 # (label, model_version, template_set_id) で正規化
  → ジャッジのテキスト側を凍結・キャッシュ

Pair (Aggregate Root)             # ペア三つ組（凍結）
  pair_id
  base   : {object_id, canonical_label, aliases[]}      # X
  target : {object_id, canonical_label, aliases[]}      # Y
  hard_negatives : [{object_id, label}]                 # K個・凍結
  difficulty_prior                                       # 候補選別専用・非公開
  difficulty_measured                                    # シード由来・公開（星評価）
  template_set_id
  status(candidate|seeded|active|retired)

SeedAsset                          # 第1層：シード画像（高コスト・永久保存）
  asset_id(=画像ハッシュ), pair_id, image_ref,
  gen_model, gen_prompt, gen_params_hash, created_at
  # 一度だけ生成。モデル交換時も再生成しない。

SeedScores                         # 第2層：シードスコア（モデル毎・安価再計算）
  pair_id, model_version, template_set_id, tau,
  scores_sorted[],                                       # 昇順rawのN個
  stats{min,p10,p50,p90,max,mean,std}, computed_at

DailyPuzzle                        # デイリー凍結（再現性の生命線）
  date, pair_id, ref_version, frozen_at
  # ref_version = (SeedScores行 + 当日始点の人間スコアスナップショット)

Submission (Aggregate Root)        # プレイヤーの一投
  submission_id, daily_puzzle_id(or pair_id), user_id,
  image_ref, stroke_log,                                 # ストロークログ＝再採点検証用
  score, percentile, raw, confidences{Cy,Cx,negs[]}, bucket,
  flags{ocr_cheat, moderation}, ref_version, created_at

TemplateBank                       # 演出Layer1（事前計算キャッシュ）
  template_id, bucket(fooled|failed|confused), locale,
  slots,                                                 # {Y}{X}{Cy}{score} プレースホルダ
  weight, mood(smug|delighted|suspicious|exasperated)
  # 参照: (bucket, locale) → バリアント乱択
```

## 4. フェーズ別ロードマップ

各フェーズに **go/no-go ゲート**を置く。ゲートを越えなければ次に進まない。

### Phase 0 — キルスイッチ検証（1〜3日）★ ビルド前に必ず
紙の上で SigLIP/CLIP を叩き、ゲームが成立するかを最小コストで確かめる。**ここで死ぬなら1日で死ぬ。**
- 手作りの擬態画像を数十枚用意（X→Y を手描き/合成）。
- 検証する3つ：(1) X→Y機構はスコアの**幅**を生むか (2) **描画で**（文字でなく）本当にモデルを騙せるか (3) 温度τは使える勾配を与えるか。
- **go/no-goゲート：** 代表的お題で「下手な絵と上手い絵がスコアで明確に割れる」が再現できるか。割れなければ**プロジェクト中止**。

### Phase 1 — 採点サービスの心臓
自前SigLIP/CLIP推論・温度・OCRガード・ハードネガティブsoftmax・描画ドメインプロンプト・raw mascore・percentile正規化。**最高リスクの技術ピースを最初に。**
- 成果物：CLIPスコアサービス（HTTP I/F）、`Pair`三つ組スキーマ、percentile擬似コード、決定性制約の実装。
- **ゲート：** 同一提出が同一スコアで再現（監査可能）＋ Phase0の手作り画像で期待通りのスコア序列。

### Phase 2 — お題プール基盤
カタログスキーマ・LLM支援タグ付けスクリプト・difficulty_prior＋ハードゲート・シード生成パイプライン（GPT流用）・2層保存・spread基準キュレーション・DailyPuzzle凍結。
- **ゲート：** 自動漏斗を通った数十ペアのうち、`spread >= floor`を満たす「良いお題」が十分な歩留まりで採れるか。**最初の数百ペアで歩留まり率を実測してから生成予算を組む。**

### Phase 3 — クライアント最小ループ（初プレイアブル）
描画キャンバス（Canvas2D/Capacitor）→ 提出 → 採点 → **テンプレバンク**コメント付きリビール → シェアカード（Pillow）。シングルプレイの一連ループ。ランキングなし。
- **ゲート：** 自分（と数人）が「もう一回やりたい」と感じるか。感じなければコアループの面白さに問題＝設計に戻る。

### Phase 4 — 非同期競争
デイリーリーダーボード・ゴースト・フレンドラダー・パーセンタイル表示・サーバー権威スコアリング＋ストロークログ再採点検証・不正対策。
- **ゲート：** 流動性のブートストラップ（シードゴースト＋パーセンタイル表示＋シェア招待）が空のラダー問題を回避できるか。

### Phase 5 — 演出強化 & 収益化
鑑定士LLMコメント層（人格プロンプト・mood表情）・ヒーローゲート・オンデマンドボタン・予算ガバナ・モデレーションパイプライン・F2P/プレミアム・シーズン制。
- **ゲート：** バズ規模を模した負荷で**LLM課金が日次上限で頭打ちになり、優雅に劣化する**ことを確認（破産しない検証）。

## 5. リスク登録簿（忖度なし）

| # | リスク | 致死性 | 緩和 |
|---|---|---|---|

## 6. コスト構造と「破産しない」設計（本家の第3教訓）
- **コストドライバ＝AI呼び出し。** 決定B（スコア＝自前CLIP）で**スコアの限界コストをほぼゼロ**にし、LLMはキャッシュ＋ヒーロー限定＋日次ガバナで上限化。バズった瞬間に1人あたり赤字、を構造で潰す。これは目標②（収益）に直結。
- **画像生成は永久保存で使い捨てない**（§1 2層保存）→ モデルローテでも生成費ゼロ。
- **課金：** F2P＋デイリー回数制限、プレミアム（練習無制限・追加試行・コスメ・広告なし・鑑定士オンデマンド無制限）。790円の衝動買い精神をモバイル到達のためF2Pに翻案。
- **UGCマーケットは作らない**（コスメのみ）→ 面が綺麗に保て、モデレーション負荷も下がる。

## 7. 成功/撤退の判断基準（先に決めておく）
- **Phase 0 で撤退**：お題がスコアの幅を生まない＝コア機構が成立しない。1日で判明、即中止。
- **Phase 3 で撤退**：自分と数人が「もう一回」と思わない＝面白さの核がない。
- **Phase 4 で要警戒**：招待→新規の成長係数が立たない＝流動性に到達しない（R1）。ここが生死の本丸。
- **sengoku-s との関係**：本ゲームは高分散・バイラル狙い、sengoku-s（放置サバイバー）は低分散・ポートフォリオ/安定収益狙い。**別の目標に効く別の賭け**。乗り換えではなく、どちらにどれだけ張るかの資源配分判断。
